# 项目状态与已知缺口

> 最后更新：2026-07-22  
> 当前定位：`0.1.0` 本地演示 / 研究原型，不是医疗器械、诊断工具、心理治疗服务或可直接公开运营的产品。

## 项目现在是什么

“听见”是一个 React + Vite 前端与 Node.js + Express 后端组成的中文心理健康觉察对话 Demo。它尝试用自然对话帮助用户整理情绪和压力，并通过本地知识规则约束普通回应与高风险分流。

仓库源码可以用于学习和继续开发，但“源码可公开”与“服务可面向公众上线”是两件事。当前工程已经具备完整的本地演示链路，尚不具备公开提供心理健康服务所需的专业审校、危机响应、隐私合规和生产运维能力。

## 已实现的主要能力

- React 单页界面，聊天记录只保存在当前浏览器标签页的 `sessionStorage`。
- Express 提供 `/api/chat`、`/api/chat-stream` 和 `/api/health`。
- 支持 OpenAI-compatible Chat Completions；真实 API key 只存在于服务端环境变量。
- 无 API key 时使用本地知识库驱动的 mock，克隆仓库后无需付费即可体验完整流程。
- 本地 JSON 知识库覆盖问题类型、机制、微干预、回应规则和安全规则。
- 高风险规则先于普通知识匹配与模型调用执行。
- 单进程内存限流和默认关闭的白名单式安全日志。
- 单元测试、知识审计、自动评测、生产 smoke 和浏览器 E2E 脚手架。

## 已知缺陷与未完成事项

### P0：公开部署前必须解决

1. **专业审校未完成。** `docs/review/knowledge-review-summary.md` 显示 45/45 条运行时知识仍为 `unreviewed`。自动规则通过不等于心理学、医学、伦理或法律审核通过。
2. **危机识别能力有限。** 当前主要依赖关键词、少量否定规则和上下文规则，存在漏报与误报；没有真人值守、升级通知或响应时限。
3. **地区化求助资源不足。** 紧急电话、热线、医院急诊和未成年人资源需要按服务地区维护并由专业人员核验。
4. **隐私与合规方案缺失。** 尚未确定服务端是否保留正文、保留期限、删除/导出机制、模型供应商的数据用途、用户同意流程和未成年人处理方式。
5. **生产安全与运维不足。** 当前没有身份认证、网关级限流、集中式配额、密钥托管/轮换、监控告警、审计存储和故障响应方案。

### P1：影响可靠性和协作

1. **来源链仍不完整。** 当前路线图记录运行时知识中只有少量条目具有明确来源引用，其余需要与证据卡、决策卡建立可追溯映射。
2. **真实模型评测不可完全复现。** 模型输出会随供应商、模型版本和时间变化；真实模型 smoke 会产生费用，也不能替代人工专业复核。
3. **覆盖面有限。** 浏览器 E2E 仍未系统覆盖无障碍、弱网、移动设备、多浏览器和长会话。
4. **接口 `mode` 值不统一。** 健康检查、普通回复、流式回复和安全回复会使用 `llm`、`api`、`mock`、`safety` 等不同值，当前前端未依赖该字段，但对外 API 合同应统一或明确文档化。
5. **工程质量门槛仍有空白。** 当前没有 lint、统一格式检查、静态类型检查和覆盖率阈值；CI 会执行无密钥预检与 Chromium E2E，但不覆盖真实模型。
6. **第三方资料与代码许可仍需确认。** `knowledge_sources/raw/` 的本地网页快照已默认排除在 Git 之外，只发布来源登记和项目摘要；若未来要提交原文，仍需逐项确认引用方式、许可证和再分发条件。项目代码许可证也尚未选择。
7. **知识上下文新路径默认关闭。** `KNOWLEDGE_CONTEXT_*` 控制的路径仍属于实验能力，默认不注入 prompt，不应被当作稳定公开接口。

### P2：后续产品化方向

- 建立经标注的多轮风险测试集和误报/漏报指标。
- 完成来源 → 证据卡 → 决策卡 → 运行时知识的可追溯发布流程。
- 增加人工升级、地区化资源和运营责任边界。
- 增加用户认证、分布式限流、监控、费用预算和模型降级策略。
- 完成可访问性、跨浏览器、移动端、性能与安全测试。

## 数据与密钥边界

- `.env`、`.env.*`（`.env.example` 除外）已被 Git 忽略；示例文件中的 `OPENAI_API_KEY` 保持为空。
- `knowledge_sources/raw/` 的第三方网页快照也不会进入 Git；本地文件保留，公开仓库只包含来源登记、摘要和空目录标记。
- 不要把密钥放入 `src/`、浏览器存储或任何 `VITE_*` 变量；Vite 会把这类变量打包进前端资源。
- `npm run check:secrets` 会扫描 Git 已跟踪和准备跟踪的文本文件，只报告路径和风险类型，不回显疑似秘密值。
- 如果密钥曾进入提交、日志、截图、Issue 或 PR，仅从文件中删除并不够，必须立即在供应商控制台撤销并轮换。
- 浏览器会话虽然只保存在 `sessionStorage`，但发送消息时正文仍会经过本项目后端；启用真实模型时还会发送给配置的模型供应商。

## 可用性判断

| 场景 | 当前结论 |
| --- | --- |
| 本地无密钥功能演示 | 可以 |
| 开源代码审阅与学习 | 完成敏感信息、第三方资料和许可证确认后可以 |
| 小范围受控内测 | 仍需完成专业审校、隐私告知和人工风险流程 |
| 面向公众提供服务 | 不可以 |
| 用于诊断、治疗或紧急危机处置 | 不可以 |

## 建议的下一里程碑

1. 完成 45 条运行时知识的专业审校，优先处理安全等级和危机回应。
2. 补齐运行时知识的来源与证据卡映射，并让严格审校校验通过。
3. 选定试用地区，完成隐私告知、数据保留规则和地区化求助资源。
4. 设计真人升级与事故响应流程，再进行受控内测。
5. 在进入公开部署前补齐生产网关、密钥管理、监控和滥用防护。

## 本地验证

推荐先运行不调用真实模型的完整检查：

```bash
npm ci
npm run preflight:local
npm run e2e:browser
```

真实模型检查需要显式配置 API key，会产生费用：

```bash
npm run eval:model
```
